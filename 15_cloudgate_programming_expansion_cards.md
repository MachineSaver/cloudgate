How to program your own expansion cards
In order to have your expansion cards working you will first have to obtain a model number from Option. Please request this with your Option point of contact before you start making hardware!

 

Once you have this number you will have to program it into the eeprom of your expansion card.

 

At boot time the CloudGate will read the model number of all the inserted expansion cards and will apply the settings defined in the devices file for this card.

 

This devices file has to be placed in the config/config/ folder at the root of the SDK file structure.

 

In order to add your board in the devices file you will have to:

 

define a new board section: "config board CG1105"
define the model number: "option model "CG1105"
 

###########
# boards #
##########
boards are identified by their "model" and optionally hwv

 
'hwv' in the config file is used as the minimum version needed. Hardware version 1.3 matches
with config file hwver 1.1 since it is newer than 1.1

 
mandatory: model
optional: hwv, description, slot_id, 
          sh_count, sh_<entries>, 
          sh_cust_count, sh_cust_<entries>,  (minimum > fw 1.26.0)
          device_count, device_<entries>
 

e.g.
config board 'Dev_board_v1'
option model "CG1108"
option description "Hardware development kit"
option slot_id "1"
option hwv "1.0"
option sh_count "0"
option device_count '0'
 

device_entries can be used to identify devices that can be found on this board,
devices follow the description as in the next paragraph
 

shell_entries can be used for various reasons: 
  * loading of kernel modules
  * setting of GPIOS
  * piping the content of the eeprom to somewhere 
 

sh_count=n marks the number of sh_<entries>, sh_<entries> are numbered from 0 to n-1, (e.g. sh_0, sh_1, ...)
this way you can specify the order the sh_entries shall be executed
 

every KEY=VALUE pair found in the eeprom (separated by \0) is put as a $KEY = VALUE pair in the environment (see man page for environ) where the shell_entries are executed
e.g.
  the eeprom contains Model=CG1234\0HWREV=1.1\0mac_addr=00:55:33:22:11:77\0\FF\FF\FF....
  than the environment will contain a $Model with a value CG1234, a $HWREV with value 1.1 and $mac_addr with value 00:55:33:22:11:77
  and you could write a sh_0
   option sh_0 "/sbin/insmod /lib/modules/2.6.35.3/your_driver.ko mac_address=$mac_addr"
 

if a customer wants to add shell_entries to an existing board (via a cust_config), sh_cust_<entries> (minimum > fw 1.26.0)
can be used for various reasons: 
  * loading of kernel modules
  * setting of GPIOS
  * piping the content of the eeprom to somewhere 
  

sh_cust_count=n marks the number of sh_cust_<entries>, sh_cust_<entries> are numbered from 0 to n-1, (e.g sh_cust_0) 
this way you can specify the order the sh_cust_entries shall be executed
 

###########
# devices #
###########
devices are identified by their "name"
 

mandatory: name, devicepath or uevent, devname, expose=0 or dev_type
optional: params, board_model
 

devicepath and devname:
  these are needed so the devman daemon is able to look for a device in sysfs when asked
  for its presence and devman does not know it (no hotplug events seen or events already passed)
  TODO this is going to be replaced by uevent triggering
 

uevent:
  some devices don't rely on devicepath and devname for lookup but trigger a uevent (and all of its sub-uevents found)
  this will fire the original hotplug event again (see kernel hotplug documentation)
   https://www.kernel.org/doc/pending/hotplug.txt
 

expose: if this device should not be exposed in device lists, mostly internal CloudGate use
dev_type: nw, serial, gpio, adc, ... 
if_type: "3g", "eth" , "wlap", "wlc"
net_type: deprecated? TODO 
 

board_model: if this device resides on one or more plug-in boards, mention the boards here, comma separated
  e.g. option board_model "CG1101,CG1102,CG1105"
 

params: (minimum fw 1.25.0.0)
  * the p_<device> parameters are values that devman will check in the environ it gets from hotplug2
  * this is done via hotplug scripts in /etc/hotplug.d/ (look for "30-inform-devman")
  * some parameters are added to the environ in the hotplug scripts (extracted from sysfs) 
  * a special case is CHOP_<parameter>, it is generated by devman from any <parameter>: (minimum > fw 1.26.0)
      trailing numbers are chopped of so they can be safely matched here, 
      consider it a simple implementation of a wild-card at the end of any environ value
      e.g DEVICENAME=mlan0 --> CHOP_DEVICENAME=mlan, DEVICENAME=wwan35 --> CHOP_DEVICENAME=wwan
